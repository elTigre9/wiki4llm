# Slash-Commands

Slash-command files are generated by `wiki4llm init` for Context Mode and Harness Mode.
They are static markdown files — the agent reads them as instructions when the user
types the command. Run Mode does not use slash-commands.

Each file is prefixed with the vault state preamble from `buildVaultPreamble()`.

---

## Context Mode

### `/wiki-map [--ask]`

```
Read the project directory (respecting project.ignore). Walk all source files.

For each significant file or module:
- Create or update an entity page in .wiki/entities/<ComponentName>.md
- Use [[wikilinks]] to link related entities

Then:
- Rewrite .wiki/map/structure.md with the current directory tree and file roles
- Rewrite .wiki/map/dependencies.md with key dependencies and versions
- Rewrite .wiki/map/entrypoints.md with main entry files and their purpose
- Rewrite .wiki/overview.md with a high-level summary of the codebase
- Rewrite .wiki/index.md with a catalog of all vault pages and one-line summaries
- Append an entry to .wiki/log.md
- Commit the vault: git add . && git commit -m "wiki4llm: map <timestamp>"

If --ask: before mapping, ask 3 clarifying questions about the codebase structure
to improve the quality of the map.
```

---

### `/wiki-bootstrap`

```
This is a new project. Read everything in specs/ (or the specsDir from .wiki4llm.json).

1. Copy each spec file to .wiki/raw/
2. Write .wiki/overview.md summarizing what the project is and does
3. Write .wiki/map/structure.md with the intended directory layout from the specs
4. Write .wiki/index.md
5. Append to .wiki/log.md
6. Commit the vault
7. Run /wiki-map to map any existing source files

If specs/ is empty or missing, say so and stop.
```

---

### `/wiki-advise`

```
The user has a question or idea (provided as the command argument).

Read the vault state preamble above. Then read:
- .wiki/overview.md
- .wiki/index.md
- Any entity or decision pages relevant to the question

Respond with:
1. How the idea fits (or conflicts with) the existing architecture
2. Risks or gaps you see
3. 2–3 concrete suggestions or alternatives

Do not write any code. Do not modify the vault. This is advisory only.
```

---

### `/wiki-build [--feature "..."] [--ask]`

```
Read the vault state preamble above. Then read:
- .wiki/overview.md
- .wiki/index.md
- Entity and decision pages relevant to the feature

Feature to build: $ARGUMENTS (or ask the user if not provided)

Plan and implement the feature. Then:
- Create or update entity pages in .wiki/entities/ for any new components
- Use [[wikilinks]] to link related entities
- Append to .wiki/log.md
- Commit the vault

If --ask: ask clarifying questions before writing any code.
```

---

### `/wiki-update [--ask]`

```
Read the vault state preamble above.

Run: git diff --name-only <last-map-commit>
(The last map commit hash is in the most recent "wiki4llm: map" entry in .wiki/log.md)

For each changed file:
- Re-read the file
- Update the relevant entity page in .wiki/entities/
- Update [[wikilinks]] as needed

Then update .wiki/map/structure.md if the directory layout changed.
Append to .wiki/log.md.
Commit the vault.

If --ask: surface any new ambiguities or contradictions you find.
```

---

### `/wiki-lint`

```
Read the vault state preamble above. Then check:

1. Every page listed in .wiki/index.md exists on disk
2. Every [[wikilink]] in any vault page resolves to an existing page
3. Every entity page has been updated within the last 30 days (flag stale ones)
4. .wiki/pending/questions.md has no unresolved - [ ] items

For each issue found, write a [HEALTH] warning to .wiki/log.md.
List all issues in your response.
Suggest which pages to prune or update.
Commit the vault after writing health warnings.
```

## Harness Mode

### `/wiki-bootstrap`

Same prompt as Context Mode `/wiki-bootstrap`, plus:

```
After completing the bootstrap, also write .wiki/pending/plan.md with a feature
checklist derived from the specs:

- [ ] feature-slug: Short description
- [ ] feature-slug: Short description

One line per feature. Use the slug: prefix format exactly.
```

---

### `/wiki-run [--no-block]`

```
Phase 1: Architect + Builder

Read the vault state preamble above. Then:
1. Read .wiki/pending/plan.md and identify the first unchecked feature (- [ ])
2. Read the relevant vault pages for that feature
3. Design the implementation: what files to create/modify, what interfaces to define
4. Implement the feature
5. Write any ambiguities to .wiki/pending/questions.md
6. Commit your code changes
7. Do NOT update the vault — that is Phase 2's job

If --no-block: make best-guess decisions on any ambiguities instead of stopping.
If no unchecked features remain, say so and stop.
```

---

### `/wiki-run --continue [--no-block]`

```
Phase 2: Mapper + Lint

Open this in a FRESH context window after Phase 1 has committed code.

Read the vault state preamble above. Then:
1. Run: git diff HEAD~1 to see what Phase 1 built
2. Update entity pages in .wiki/entities/ to reflect the changes
3. Update .wiki/map/structure.md, .wiki/map/dependencies.md, .wiki/map/entrypoints.md
4. Update .wiki/index.md
5. Read .wiki/pending/questions.md — resolve or escalate each open item
6. Check off the completed feature in .wiki/pending/plan.md (change - [ ] to - [x])
7. Run a vault health check (orphan pages, broken wikilinks, stale claims)
8. Append to .wiki/log.md
9. Commit the vault

If --no-block: resolve all questions with best-guess, do not pause.
```
